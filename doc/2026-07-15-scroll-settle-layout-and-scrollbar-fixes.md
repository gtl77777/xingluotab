# 滚动停止重绘、布局偏移与滚动条体验优化

## 文档信息

- 日期：2026-07-15
- 对应提交：`9cda638 fix: stabilize scroll settling and app layout`
- 涉及范围：空间列表虚拟滚动、主题表面样式、左右侧栏布局状态、全局滚动条、浏览器回归测试

## 背景

本次会话主要处理了三个互相关联的体验问题：

1. 列表滚动过程比较流畅，但滚动停止后会出现大范围重绘、卡片和图标闪动。
2. 侧栏收纳后，从 About 页面返回空间页面时，主内容会短暂向右偏移。
3. 原生滚动条视觉过重，与当前界面风格不统一。

优化目标不是简单增加过渡动画，而是在不重新引入滚动卡顿的前提下，让滚动停止、路由切换和滚动条交互都保持稳定。

## 一、滚动停止后的大范围闪动

### 问题表现

- 高速滚动时使用轻量预览，滚动过程本身没有明显卡顿。
- 滚动停止后，当前视口内多个分组会同时发生视觉变化。
- 图标、卡片背景和分组背景看起来会先加深，再恢复正常。
- 专业主题相对不明显；Mica、Aurora、Paper 等亮色主题以及不同强调色下更容易观察到。
- 早期的透明度揭示动画会让分组产生“一抖一抖”的感觉。

### 根因

#### 1. 静态预览与完整交互层一次性切换

虚拟列表在滚动时使用轻量静态预览，以避免同时创建大量拖拽、菜单、图片和测量逻辑。原实现会在虚拟滚动器判定滚动停止后，将视口内的静态预览一次性替换成完整交互组件。

这个瞬间会集中发生：

- React 组件挂载；
- 拖拽节点和监听器注册；
- favicon 图片解码和回退源切换；
- 虚拟列表重新测量；
- 卡片阴影、模糊和背景效果重新绘制。

因此，肉眼看到的不是单个图标刷新，而是整个可见区域共同变化。

#### 2. 静态卡片与真实卡片的主题选择器不一致

静态预览卡片和完整交互卡片使用了不同的数据属性，部分视觉主题只命中了其中一层：

- Mica、Aurora 的 `backdrop-filter` 命中范围不一致；
- Paper 的卡片阴影只应用于其中一种卡片；
- OLED 和 Zen 模式下也存在表面规则入口不统一的问题。

两层即使尺寸完全相同，背景、阴影或模糊效果不同，切换时仍然会表现为整组颜色先变深再恢复。

#### 3. 透明度动画放大了切换感

旧方案在真实分组挂载时执行短暂的透明度动画。它不能减少组件挂载和绘制成本，反而让静态层与真实层的差异在一段时间内持续可见，因此被感知为抖动。

### 最终方案

采用“冻结预览、渐进准备、原子揭示”的三阶段方案。

#### 阶段 1：滚动期间只显示静态预览

- 保留虚拟列表和现有滚动性能优化。
- 静态预览只绘制当前需要看到的卡片窗口。
- 不在滚动过程中创建完整拖拽和菜单交互树。
- 固定虚拟行高度，避免预览阶段产生布局跳动。

#### 阶段 2：滚动停止后在隐藏层渐进准备

- 先预加载当前窗口可能显示的 favicon。
- 按“视口内优先、距离视口中心更近优先”的顺序处理分组。
- 每个动画帧只激活一个分组，分散 React 挂载和测量开销。
- 真实交互层准备期间保持 `visibility: hidden`。
- 上方继续覆盖冻结的静态预览，因此用户不会看到半完成状态。
- 缓冲预览层不参与拖拽命中，避免隐藏交互层与预览层同时响应。

#### 阶段 3：全部提交后一次性揭示

- 所有可见交互分组准备完成后，再等待两个 `requestAnimationFrame`。
- 确认 React 已提交、浏览器已完成必要布局后，统一移除预览覆盖层。
- 不使用透明度、位移或宽度动画。
- 静态层与真实层几何尺寸、背景、边框、阴影和模糊效果保持一致，因此切换不可感知。

### 主题表面统一

静态卡片和真实卡片统一增加：

```html
data-record-tab-surface="true"
```

所有与卡片表面有关的主题规则都基于该属性处理，包括：

- Mica 背景模糊；
- Aurora 背景模糊；
- Paper 阴影；
- OLED 阴影覆盖；
- Zen 模式卡片阴影。

这样不论当前是滚动预览层还是完整交互层，最终像素效果都一致。

### 性能取舍

该方案不会重新引入原来的滚动卡顿：

- 滚动热路径仍然只渲染轻量静态预览；
- 完整交互层在停止后分帧挂载，不会集中占用主线程；
- 缓冲期短暂保留两层 DOM，但只覆盖虚拟窗口中的少量可见分组；
- 原子揭示只是可见性切换，不执行逐项动画；
- 不监听滚动位置来驱动自定义动画。

## 二、从 About 返回空间页面时主内容右移

### 问题表现

右侧会话栏或左侧空间栏收纳后，进入 About，再返回空间页面，主内容会短暂向右移动，随后才恢复正确位置。

### 根因

侧栏折叠状态原先由不同路由组件分别维护：

- 空间页面挂载时先使用本地默认值；
- 随后异步读取 `chrome.storage.local`；
- 读取完成后再修改左右侧栏宽度。

从 About 返回空间页面会重新挂载空间页面。第一帧使用默认展开状态，存储读取完成后又切换为收纳状态，因此主区域宽度和横向位置发生二次变化。

### 处理方案

新增全局 `LayoutSettingsProvider`，在路由渲染之前统一加载并持有：

- `isSidebarCollapsed`；
- `isSessionBarCollapsed`。

具体行为：

- 布局设置未加载完成时不渲染可能产生错误宽度的路由页面；
- AppShell、空间侧栏、空间页面和会话栏共享同一份内存状态；
- 路由切换不会销毁这份状态；
- 点击折叠按钮时先进行内存中的乐观更新，再写入存储；
- 写入失败时回读存储并恢复有效状态；
- 监听存储变化，使其他页面或扩展上下文的设置修改可以同步；
- 解析失败时保留最后一次有效布局，不退回默认宽度。

同时给主内容和会话栏增加测试定位属性：

```html
data-space-main="true"
data-session-bar="true"
```

用于持续检测路由往返过程中的横向位置和宽度变化。

## 三、滚动条视觉改造

### 采用方案

最终采用 Fluent 风格的极简细滚动条：

- 轨道透明；
- 可操作区域为 `10px`；
- 内部实际可见宽度为 `6px`；
- 默认使用中性的弱化前景色；
- 悬停只提高对比度，不改变宽度；
- 按下拖动时才使用当前主题强调色；
- 滑块最小高度为 `32px`；
- 不添加颜色、宽度或位移动画。

### 性能说明

滚动条完全由 CSS 和浏览器原生滚动机制实现：

- 没有 JavaScript 滚动监听；
- 没有自绘滚动层；
- 没有模拟惯性或滚动动画；
- 不改变现有虚拟列表逻辑；
- 不会因悬停导致内容宽度变化。

Chrome、Edge、Brave 等 Chromium 浏览器通过 `::-webkit-scrollbar` 系列伪元素实现；Firefox 使用 `scrollbar-color` 和 `scrollbar-width`。在系统强制高对比度模式下不覆盖系统滚动条，以保留可访问性。

## 七、后续维护注意事项

1. 修改静态预览或真实标签卡片样式时，应同时通过 `data-record-tab-surface="true"` 维护公共表面效果。
2. 不要在滚动停止时重新引入逐分组透明度或位移动画。
3. 新增昂贵的交互组件时，应继续放在隐藏交互层的渐进准备阶段，不要加入滚动热路径。
4. 左右侧栏宽度状态应继续由 `LayoutSettingsProvider` 统一管理，不要在路由页面中复制本地折叠状态。
5. 修改虚拟列表、主题样式或布局状态后，至少运行：

   ```powershell
   npm run compile
   npm test
   node scripts/test-scroll-settle.mjs --port <调试端口>
   ```

6. 发布前应重新生成 Chrome 和 Edge 两个压缩包，避免安装包仍包含旧样式。
