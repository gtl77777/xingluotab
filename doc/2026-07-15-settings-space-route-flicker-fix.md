# 设置与空间页面切换闪烁修复

## 文档信息

- 日期：2026-07-15
- 实现提交：`d8f79a4 fix: prevent route transition flicker`
- 涉及范围：设置页、空间页、空间侧栏、全局设置状态、空间页缓存、真实浏览器回归测试

## 一、问题表现

此前已经修复了滚动停止后的卡片闪动，以及从 About 页面返回空间页时因侧栏折叠状态异步恢复造成的横向偏移。

本次发现另一条路由切换闪烁链路：

1. 从空间页点击“设置”时，左侧空间列表和设置内容会短暂刷新。
2. 从设置页点击某个空间返回时，空间页会先出现空白或加载状态，再恢复实际分组和标签内容。
3. 使用自定义 Logo 时，侧栏重新挂载会让 Logo 短暂恢复为默认图标。
4. Mica、Aurora 等带背景模糊的主题会放大侧栏重新绘制的视觉变化。

该问题与滚动虚拟化恢复无关，也不是昨天的侧栏宽度修复失效。宽度和位置已经稳定，本次闪烁来自组件及数据生命周期重置。

## 二、根因分析

### 1. 空间页和设置页分别创建侧栏

原路由结构中：

- `SpacePage` 内部渲染一棵 `SpaceSidebar`；
- 设置、同步和 About 页面通过 `SpaceSidebarPageLayout` 渲染另一棵 `SpaceSidebar`。

在空间页与设置页之间切换时，React 会卸载当前路由组件，再挂载另一套路由组件。虽然两边看起来是同一个侧栏，但实际 DOM 节点、拖拽上下文和内部状态全部重新创建。

### 2. 新侧栏第一帧使用空空间列表

`SpaceSidebarPageLayout` 原先以空数组初始化空间列表：

```ts
const [spaces, setSpaces] = useState<SpaceSummary[]>([]);
```

组件挂载后的 `useEffect` 才调用 `getSpaceList()`。因此至少存在一个空间列表为空的渲染阶段，然后才填充真实数据。

### 3. 设置页第一帧使用默认值

`SettingsPage` 原先以 `userSetting = null` 启动，并在挂载后异步调用 `getUserSetting()`。首帧会先渲染禁用控件和默认选项，再替换为真实设置。

### 4. 返回空间页时页面状态完全重置

`SpacePage` 被重新挂载后会恢复以下默认状态：

- 空空间列表；
- 当前空间为 `null`；
- `loading = true`；
- 空会话标签列表；
- 默认卡片视图、排序、禅模式和折叠分组设置。

随后页面再次读取空间、设置和当前浏览器标签，导致“空白或加载帧 → 实际内容”的明显切换。

### 5. 旧回归测试只检查几何稳定性

原真实浏览器测试覆盖了 About 返回空间页后的主区域位置、宽度和会话栏宽度，但没有检查：

- 设置页路由；
- 侧栏 DOM 是否被替换；
- 空间行是否短暂清空；
- 返回空间时是否出现加载帧；
- 缓存内容是否在首帧可见。

因此旧测试可以在几何尺寸完全稳定的情况下通过，却无法发现内容重建造成的视觉闪烁。

## 三、修复方案

### 1. 将空间侧栏提升到路由外常驻

`SpaceSidebarPageLayout` 现在包裹整棵 `Routes`：

```tsx
<AppShell>
  <SpaceSidebarPageLayout>
    <Routes>{/* 空间、设置、同步、About */}</Routes>
  </SpaceSidebarPageLayout>
</AppShell>
```

路由切换只替换右侧页面内容，左侧空间侧栏保持同一组件实例和同一 DOM 节点。

收益包括：

- 空间列表不会回到空数组；
- 自定义 Logo 不会重新异步加载；
- 侧栏拖拽上下文不会反复创建；
- Mica、Aurora 背景模糊层不再因路由切换重建；
- 折叠状态、宽度和侧栏内容同时保持稳定。

侧栏自己的拖拽 `DndContext` 只包裹侧栏，不包裹右侧空间页，避免与空间页内部的分组和标签拖拽上下文互相干扰。

### 2. 全局持有完整用户设置

`LayoutSettingsProvider` 从只保存两个折叠布尔值扩展为持有完整 `UserSetting`。

它现在负责：

- 应用存储中的完整初始设置；
- 监听 `chrome.storage.local` 变化；
- 乐观更新内存设置；
- 串行执行设置写入，减少并发覆盖风险；
- 写入失败时从存储恢复有效值；
- 为设置页、主题、Logo、侧栏布局和空间页布局提供同一数据源。

### 3. 设置页首帧直接使用真实设置

`SettingsPage` 不再自行以 `null` 初始化并重新读取存储，而是直接使用已经准备好的全局 `userSetting`。

因此进入设置页时：

- 不再出现禁用控件阶段；
- 不再从默认语言或默认主题选项跳到真实值；
- Logo 预览和外观设置首帧即正确。

### 4. AppShell 和 SpaceSidebar 复用全局设置

`AppShell` 直接从 Provider 获取主题、视觉主题和强调色，不再建立第二套设置监听状态。

`SpaceSidebar` 直接读取 `userSetting.logoDataUrl`，不再在每次挂载后单独异步加载 Logo。

### 5. 缓存最后一次有效空间页面状态

`SpacePage` 增加轻量内存缓存：

- 以空间 ID 缓存最后一次有效 `SpaceState`；
- 缓存最后一次会话标签列表；
- 重新进入已经访问过的空间时同步恢复缓存；
- 同时在后台重新读取存储和当前浏览器标签，保证数据最终是最新的；
- 缓存不保存短暂提示文本，避免返回页面时重放旧通知。

这不是让隐藏的空间页继续运行。空间页离开路由后仍会正常卸载，不会继续注册拖拽、滚轮或浏览器标签监听；重新进入时只复用最后一次有效数据作为首帧内容。

### 6. 保持禅模式行为

侧栏提升到路由外后，布局根据当前路由和全局 `zenMode` 决定是否显示侧栏：

- 空间页进入禅模式时隐藏侧栏；
- 离开空间页进入设置、同步或 About 时恢复侧栏；
- 返回禅模式空间时首帧保持隐藏，不出现侧栏闪现。

### 7. 删除空间页内重复的侧栏功能

空间页不再维护仅供内部侧栏使用的重复代码，包括：

- 新建空间入口；
- 导入空间和 Toby 文件输入；
- 空间图标对话框；
- 空间导出和删除侧栏处理；
- 第二棵空间侧栏。

空间重命名仍保留在空间页标题区域；公共侧栏中的新建、导入、图标、导出、删除和排序统一由常驻布局处理。

## 八、后续维护注意事项

1. 空间侧栏应继续保持在路由外，不要重新放回 `SpacePage` 或各个设置类页面。
2. 新增全局设置字段时优先复用 `LayoutSettingsProvider`，避免页面各自以默认值启动后异步跳变。
3. 空间缓存只用于首帧稳定，后台存储刷新仍必须保留。
4. 不要通过隐藏并常驻整个 `SpacePage` 来缓存页面，这会保留不必要的全局事件和浏览器标签监听。
5. 修改路由布局后，应同时检查几何稳定性和内容稳定性；只检查元素宽度无法发现空列表或加载帧。
6. 发布前至少执行：

   ```powershell
   npm run compile
   npm test
   npx wxt zip
   npx wxt zip -b edge
   node scripts/test-scroll-settle.mjs --port <调试端口>
   ```
